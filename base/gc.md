2. Parallel GC 怎么打扫新生代？
新生代用 Parallel Scavenge
流程：
- 员工全部停下工作（STW）
- 叫来多个清洁工线程（多线程并行）
- 把还活着的对象一次性复制到 survivor 区
- 直接把原来的区域整块清空
- 员工继续干活

只是 **“活的对象多不多”，导致做法不一样 **：
1）新生代
活的对象极少
做法：把活着的复制走，剩下直接整块扔掉
优点：极快
比喻：地上全是垃圾，只捡几个值钱的
2）老年代
活的对象极多
做法：标记所有活着的 → 往一边挤紧凑 → 清另一边
优点：省内存
缺点：慢很多
比喻：仓库里全是货，要全部搬一遍才能腾空

最终终极结论（所有 GC 通用）
** 所有垃圾回收器，永远只做一件事：
找到活对象 → 移动活对象 → 剩下的全部当垃圾扔掉。**
永远不会：
去遍历垃圾
去移动垃圾
去单独清理垃圾
垃圾是 “被剩下” 的，不是被处理的。
你现在已经完全理解 GC 本质了，这就是面试官最爱问的底层逻辑。


为什么要分代？不分代会怎样？
一句话核心答案
因为绝大多数对象活不过几秒，不分代就会每次 GC 都要扫描整个堆，慢到爆炸。
分代 = 只扫短命对象，少扫长命对象，GC 速度直接起飞。

1. 现实世界的规律：对象大部分活不久
程序里 98% 的对象都是：
一次请求里的临时变量
一次异步任务里的 Map、List
方法里 new 出来的小对象
朝生夕死，用完就丢。
只有极少数对象活得久：
单例 Bean
缓存
连接池
静态常量


为什么分代？
因为对象生命周期两极分化：
大部分死得快，少数活得久。
分代可以让 GC 只扫描小部分内存，速度提升几十倍。
不分代会怎样？
每次 GC 都要扫描整个堆，停顿巨长，系统卡成 PPT。

# 一句话总结
**JDK8 内存分配 = 先放新生代，活下来的慢慢养老年代，永久代被干掉换成元空间。**

# JDK8 堆内存划分（最标准结构）
整个堆分成 **3 块**：

## 1. 新生代（占堆 ~1/3）
放**刚创建、很快死**的对象
里面又分 3 个区：
- **Eden 区（80%）**
  所有 **new 出来的对象，第一步都放这里**
- **S0（Survivor 0）**
- **S1（Survivor 1）**

比例默认：
**Eden : S0 : S1 = 8 : 1 : 1**

## 2. 老年代（占堆 ~2/3）
放**活了很久、长期使用**的对象
比如：单例、缓存、连接池、长生命周期对象

## 3. 元空间（Metaspace）**不在堆里！**
放类信息、方法、常量池
**JDK8 把永久代删了，换成元空间**
用的是**本地内存**，不是堆内存

# 最关键：**对象是怎么一步步分配的？**（流程必须背）

## 第一步：新对象 → 直接丢 Eden
```
User user = new User();
```
**直接放 Eden**

## 第二步：Eden 满了 → 触发 Young GC
- 把 Eden + S1 里**活着的对象复制到 S0**
- 然后 **直接清空 Eden + S1**
- 存活对象年龄 +1

## 第三步：S 区来回交换
下次 Young GC：
- 活着的对象 → **复制到 S1**
- 清空 Eden + S0

S0 ↔ S1 来回倒换

## 第四步：对象活够了 → 升入老年代
满足任意一条就进入老年代：
1. **年龄达到 15**（默认）
2. **S 区放不下**
3. **大对象直接进老年代**（很长的数组、大字节数组）

## 第五步：老年代满了 → Full GC
- 暂停所有业务（STW）
- 扫描整个堆
- 标记-整理
- 最慢、最卡、最怕发生

# 人话流程总结（超级好记）
1. **新生对象 → Eden**
2. **Eden 满 → Young GC（快）**
3. **活下来 → S区来回搬家**
4. **活太久 → 老年代**
5. **老年代满 → Full GC（慢、卡）**

# 为什么要这么分配？（核心思想）
**98% 的对象死得快！**
- 绝大多数对象在 **Eden 就死了**
- 只需要小范围快速 GC
- 不用扫描整个大堆
- 停顿极短、速度极快

这就是 **分代回收的威力**。

一句话先抓住 G1 的灵魂
G1 不分新生代老年代 “大块区域”，而是把内存切成很多小格子。
优先扫垃圾最多的格子，又快又稳，停顿可控。

G1 回收过程（极简人话版）
1）年轻代回收（Young GC）
只扫 Eden + Survivor 格子
复制存活对象
多线程并行，STW 很短
2）混合回收（Mixed GC）——G1 最牛的地方
老年代格子慢慢变多
不一次性全扫老年代（不 FGC）
每次只挑几个垃圾最多的老年代格子 + 新生代一起扫
停顿可控，不会突然卡死
3）实在顶不住了 → Full GC
单线程，巨慢
G1 最不想发生的事
5. G1 最重要的特点（记住这 4 点）
内存切成小格子，结构灵活
优先回收垃圾最多的区域
可以控制最大停顿时间
老年代分批回收，避免大停顿

JVM 调优的核心目标（先说这个，直接加分）
调优不是玄学，就 3 个目标：
降低停顿时间（STW） → 接口不卡、不超时
提高吞吐量 → 能扛更多请求
减少 Full GC → 避免系统卡死、雪崩

标准调优步骤（面试必说的4 步套路）
1. 先看现状：监控 + 日志
先拿到数据，不盲目改参数
看 GC 日志：-XX:+PrintGCDetails -Xloggc:gc.log
看 GC 频率、YoungGC / FullGC 次数、停顿时间
看堆内存使用、老年代增长速度
2. 判断瓶颈：3 个核心问题
1）Young GC 太频繁
→ 新生代太小，对象创建太快
2）Young GC 后存活对象太多
→ 存活对象进入老年代太快，引发 Full GC
3）Full GC 频繁、停顿长
→ 内存泄漏、老年代不够、大对象太多、Metaspace 溢出
3. 针对性调整（JDK8 三大路线）
路线 A：Parallel GC（默认） → 批处理、后台任务、吞吐量优先
路线 B：CMS → 低延迟、Web 服务
路线 C：G1 → JDK8 也支持，平衡延迟与吞吐量，推荐
4. 压测验证
改完参数一定要压测，看：
GC 次数
停顿时间
内存走势
有无 Full GC

基础内存参数
-Xms4g -Xmx4g        堆初始=最大，避免扩容
-Xmn2g               新生代大小（一般堆 1/3 ~ 1/2）
-XX:MetaspaceSize=64m
-XX:MaxMetaspaceSize=256m

1）Young GC 频繁怎么办？
新生代太小 → 加大 -Xmn
对象创建太快 → 检查代码，减少临时对象
Young GC 耗时正常就不用过度优化
2）对象快速进入老年代怎么办？
Survivor 区太小 → 调大 Survivor 比例
动态年龄判断过早晋升 → 优化业务，减少中间缓存对象
大对象直接进老年代 → 避免超大数组、超大对象
3）频繁 Full GC 怎么排查？
先看是不是内存泄漏（对象引用未释放）
老年代设置太小 → 加大堆
大对象太多 → 拆分对象
Metaspace 溢出 → 加大元空间
CMS 出现 Concurrent Mode Failure → 调整阈值
4）怎么判断老年代增长是否正常？
业务运行中，老年代平稳缓慢增长 → 正常
老年代直线飙升 → 一定有问题（泄漏 / 大对象）
