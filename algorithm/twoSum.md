# 两数之和（LeetCode 1）

## 题目

给定一个整数数组 nums 和一个目标值 target，找出数组中和为 target 的两个数，返回它们的**索引**。

假设每个输入只有一个答案，且同一个元素不能使用两次。

## 思考过程

### 1. 暴力解怎么做？

双层 for 循环，逐对检查：

```java
for (int i = 0; i < n; i++)
    for (int j = i+1; j < n; j++)
        if (nums[i] + nums[j] == target) return {i, j};
```

时间 O(n²)，瓶颈在哪？→ 内层循环本质是在找 `target - nums[i]` 是否存在。

### 2. 怎么加速"查找某个固定值"？

查找最快的数据结构 → 哈希表 O(1)。

### 3. 一次遍历会不会漏掉结果？

最开始 Map 是空的，确实会"漏掉"前面的数。但两数之和是互相的——如果 a + b = target，遍历到 a 时找不到 b，那遍历到 b 时一定能找到 a（因为 a 已经存进 Map 了）。

### 4. Map 会不会存在索引覆盖的问题？

会覆盖，但不影响正确性：

- **如果重复值是解**：先查再存，第二个进来时先查到第一个，直接返回，还没来得及覆盖
- **如果重复值不是解**：覆盖了也无所谓，题目说只有一个答案，用哪个索引都行

### 5. 为什么不用排序 + 双指针？

题目要返回**索引**，排序会打乱原始位置。虽然可以额外记录原始索引，但比哈希表方案复杂且容易出错（重复值覆盖索引的问题更难处理）。

## 思维链总结

```
要返回索引 → 不能排序
暴力 O(n²) → 瓶颈是"查找配对值"
加速查找 → 哈希表 O(1)
一次遍历会漏？ → 不会，两数互为配对，后面能找到前面
覆盖问题？ → 先查再存，是解就提前返回了，不是解覆盖也无所谓
```

## 最终代码

```java
class Solution {
    public int[] twoSum(int[] nums, int target) {
        Map<Integer, Integer> indexMap = new HashMap<>();

        for (int i = 0; i < nums.length; i++) {
            int complement = target - nums[i];
            // 先查：配对值之前出现过吗？
            if (indexMap.containsKey(complement)) {
                return new int[]{i, indexMap.get(complement)};
            }
            // 再存：没找到配对，把自己存进去等别人来找
            indexMap.put(nums[i], i);
        }
        return new int[]{};
    }
}
```

## 复杂度

| 维度 | 复杂度 | 说明 |
|------|--------|------|
| 时间 | O(n) | 一次遍历，哈希表查找 O(1) |
| 空间 | O(n) | 最坏情况所有元素都存进 Map |

## 易错点总结

1. **先存再查（错误）vs 先查再存（正确）**：先存再查会导致自己和自己配对，比如 `[3, 2, 4]` target=6，i=0 时存入 3，然后查 complement=3 找到自己，返回 `[0, 0]` 错误
2. **用排序 + 双指针**：返回的是索引，排序后索引乱了；重复值还会导致 Map 中索引覆盖
3. **忘记处理重复值**：其实不需要特殊处理，"先查再存"天然解决了这个问题
4. **return 的索引顺序**：题目一般不要求顺序，`[i, map.get(complement)]` 或反过来都行

## 通用套路

| 题目特征 | 选择方案 |
|---------|---------|
| 找两个数，返回**索引** | 哈希表（不能排序） |
| 找两个数/三个数，返回**值的组合** | 排序 + 双指针 |
| 在数组中查找"是否存在某个值" | 哈希表 |
