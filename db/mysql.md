### 存储引擎
  + myisam 纯读场景，写性能极差，表锁，索引和数据存储分离
  + innodb 90%使用，支持ACID事务，回滚，行级锁，聚簇索引，数据和主键索引存在一起，主键查询极快
#### innodb底层
    1. 使用redo log（持久性）和undo log（原子性）支撑事务的实现
    2. 事务执行时，先写 Redo Log 缓冲，再刷盘（可配置 innodb_flush_log_at_trx_commit：1 = 事务提交即刷盘，最安全；0 = 每秒刷盘，性能高但可能丢 1 秒数据）；服务器崩溃后，InnoDB 重放 Redo Log，恢复未刷到数据文件的修改。
    3. 事务修改数据前，先将旧数据写入 Undo Log；事务回滚时，从 Undo Log 恢复数据；同时 Undo Log 也是 MVCC（多版本并发控制）的基础 —— 读操作通过 Undo Log 读取历史版本，避免阻塞写操作。
    4. buffer pool缓冲池，命中直接读取缓存, 提高查询速度，内存的50%-70%
#### MVCC
  1. 在没有 MVCC 的传统锁机制中：读操作需要加读锁，写操作需要加写锁，读写互斥。当有事务在写数据时，其他事务的读操作必须等待，严重影响并发性能。换句话说，MVCC解决的是读写冲突，并不解决并发写
  2. MVCC 的思路是：写数据时创建新版本，读数据时读取旧版本，从而实现“读不加锁，读写不冲突”。
  3. MVCC（Multi-Version Concurrency Control）是 InnoDB 实现读不阻塞写、写不阻塞读的核心机制，本质是「为数据行维护多个版本，不同事务看到不同版本的数据」，既保证并发性能，又能满足事务隔离级别（尤其是 RR 可重复读）。
  4. MVCC 是 InnoDB 实现高并发的核心，靠「隐藏列 + Undo Log 版本链 + Read View」实现；本质是让不同事务看到数据的不同版本，快照读无锁，大幅提升并发；
  5. RR 隔离级别下，Read View 仅生成一次，保证 “可重复读”；RC 级别每次查询生成，仅保证 “读已提交”。
  6. 一句话记忆 MVCC = 数据行隐藏列（标记版本） + Undo Log（存历史版本） + Read View（判断可见性） → 读写不阻塞，并发更高效。
  7. 版本链会不断增长，占用大量存储空间。InnoDB 有一个后台线程（purge 线程）定期清理那些旧版本。判断条件：当一个版本的 DB_TRX_ID 小于当前所有活跃事务的最小 min_trx_id（即该版本的事务已经提交且所有事务都已经不再需要它），并且该版本没有被任何事务的 Read View 引用时，就可以安全删除。
   
#### 并发写
  1. InnoDB 严格遵循「先锁后写」：先加排它锁（行锁），MVCC 记版本（支持回滚 / 快照读）；

### 其他
  + 布尔类型使用tinyint(1) 0，1来存储
  + 手机号推荐使用varchar来存，固定长度，无需计算，前导0。通常查询会需要通过号段来查like 138%；可能会有加密存储的要求，加密后是一串字母或者数字。长度要根据加密算法的长度来进行计算 128或256
  + CHAR(M) 和 VARCHAR(M) 的 M 都代表能够保存的字符数的最大值，无论是字母、数字还是中文，每个都只占用一个字符。
  + VARCHAR(100)和 VARCHAR(10)都是变长类型。，VARCHAR(100) 会消耗更多的内存。这是因为 VARCHAR 类型在内存中操作时，通常会分配固定大小的内存块来保存值，即使用字符类型中定义的长度。例如在进行排序的时候，VARCHAR(100)是按照 100 这个长度来进行的，也就会消耗更多内存。

### 流程
client -> 连接器（鉴权，管理）-> 查询缓存（命中立即返回，8.0后移除）-> 分析器 -> 优化器 -> 执行器 -> 存储引擎
- MySQL 8.0 移除查询缓存的核心原因是其适配场景极窄且代价高：首先，表中任意数据修改都会清空该表的所有缓存，高并发写场景下命中率极低，还会引发锁竞争；其次，缓存 Key 要求 SQL 完全一致，动态参数化查询几乎无法命中；最后，其收益可被应用层缓存（Redis）、InnoDB Buffer Pool 替代，维护成本却极高。因此在读写混合的现代业务中，查询缓存弊大于利，最终被移除。


## 索引
  1. b树和b+树的区别：b树所有节点上都存数据，b+树只在叶子节点上存数据，非叶子结点只存索引，能存更多索引， b+树的高度低，减少io；叶子节点使用双向链表链接，范围查询更快。
  2. innodb默认主键是聚簇索引 b+树实现，非叶子节点存主健索引，叶子结点存主键+真实数据。二级索引，非叶子节点存索引，叶子结点存索引+真实数据的主键，所以二级索引查询非索引字段时，需要进行回表
  3. 如果通过二级索引查询出来的主键比较分散，又必须要回表时，mysql通过mrr先对主键进行排序，再回表，减少io
  4. 最佳实践：索引覆盖，尽量避免select * 的出现，jpa可以通过projection实现索引覆盖
  5. 红黑树：是二叉树，b+是多路树。java TreeMap， hashMap中的链表大于8转红黑树
  6. b+树平衡：B + 树的平衡核心是「所有叶子节点在同一层」，无论插入 / 删除多少数据，树的高度始终保持极低（百万级数据仅 3 层），这是靠「节点分裂」和「节点合并」两大操作实现的。叶子节点的容量由innodb页大小决定，16kb
  7. 红黑树平衡，先染色在旋转，保持左右树的高度差，遵循五大规则
  8. 其他索引：hash，BST（最坏退化成链表）,AVL（高度差不超过1平衡）,红黑树（大致平衡），
