# MySQL 核心

## 一、MySQL 基础特性

### 1. 核心定位

关系型数据库，具备强事务支持、易运维、开源生态完善、社区活跃等核心优势。  

### 2. 时间格式对比


| 类型        | 时区特性                     | 对比 PG/GaussDB                              |
| --------- | ------------------------ | ------------------------------------------ |
| datetime  | 无时区属性，时区处理完全交由应用程序       | -                                          |
| timestamp | 存储时转换为 UTC 时间，查询时按会话时区返回 | PG/GaussDB：timestamp（无时区）、timestamptz（有时区） |


### 3. 特殊字段存储规范

- **布尔类型**：使用 `tinyint(1)`，以 0/1 存储；  
- **手机号**：推荐 `varchar` 类型（支持前导 0、号段模糊查询 `like 138%`，适配加密后字符存储，长度按加密算法调整为 128/256）；  
- **CHAR(M) vs VARCHAR(M)**：  
  - M 均表示字符数最大值（中文/字母/数字均算 1 个字符）；  
  - VARCHAR 为变长类型，但 `VARCHAR(100)` 比 `VARCHAR(10)` 消耗更多内存（排序/内存操作按定义长度分配）。

## 二、存储引擎（核心：InnoDB）

### 1. 引擎对比


| 引擎     | 适用场景     | 核心特性                                   |
| ------ | -------- | -------------------------------------- |
| MyISAM | 纯读场景     | 写性能差、表锁、索引与数据存储分离                      |
| InnoDB | 90% 业务场景 | 支持 ACID 事务/回滚、行级锁、聚簇索引（主键+数据共存，主键查询极快） |


### 2. InnoDB 底层核心

#### （1）事务支撑：Redo Log + Undo Log


| 日志类型     | 核心作用                             | 关键配置                                                              |
| -------- | -------------------------------- | ----------------------------------------------------------------- |
| Redo Log | 保证事务持久性，服务器崩溃后恢复未刷盘数据            | `innodb_flush_log_at_trx_commit`：1=提交即刷盘（最安全）；0=每秒刷盘（高性能，可能丢1秒数据） |
| Undo Log | 保证事务原子性（回滚），作为 MVCC 基础（存储数据历史版本） | -                                                                 |


#### （2）Buffer Pool 缓冲池

- 核心作用：缓存数据页/索引页，命中时直接内存读取，提升查询速度；  
- 配置建议：占用物理内存的 50%-70%。

#### （3）MVCC（多版本并发控制）


| 核心要点   | 说明                                                      |
| ------ | ------------------------------------------------------- |
| 解决问题   | 读写冲突（读不加锁、读写不冲突），不解决并发写问题                               |
| 核心思路   | 写数据创建新版本，读数据读取旧版本，实现“读写不阻塞”                             |
| 底层实现   | 隐藏列（标记版本） + Undo Log（历史版本链） + Read View（版本可见性判断）        |
| 隔离级别适配 | RR（可重复读）：Read View 仅生成一次；RC（读已提交）：每次查询生成 Read View      |
| 版本清理   | purge 后台线程清理无引用的旧版本（DB_TRX_ID < 所有活跃事务最小 min_trx_id）    |
| 一句话总结  | MVCC = 数据行隐藏列 + Undo Log 版本链 + Read View → 读写不阻塞，提升并发效率 |


#### （4）并发写处理

InnoDB 遵循「先锁后写」：先加行级排他锁，再通过 MVCC 记录版本（支持回滚/快照读）。  

### 3. MySQL 执行流程

### `client → 连接器（鉴权/管理）→ 查询缓存（8.0后移除）→ 分析器 → 优化器 → 执行器 → 存储引擎`

- **查询缓存移除原因**：  
  1. 高并发写场景命中率极低（任意数据修改清空该表缓存）；
  2. 缓存 Key 要求 SQL 完全一致，动态参数化查询几乎无法命中；
  3. 收益可被应用层缓存（Redis）、InnoDB Buffer Pool 替代，维护成本极高。

## 三、索引核心（InnoDB 基于 B+ 树）

### 1. 基础索引结构对比

#### （1）B 树 vs B+ 树


| 特性   | B 树     | B+ 树                                   |
| ---- | ------- | -------------------------------------- |
| 数据存储 | 所有节点存数据 | 仅叶子节点存数据，非叶子节点仅存索引（存储更多索引，降低树高度，减少 IO） |
| 范围查询 | 效率低     | 叶子节点双向链表链接，范围查询更快                      |


#### （2）常见索引结构平衡机制


| 结构   | 类型  | 平衡机制                                     | 适用场景                                     |
| ---- | --- | ---------------------------------------- | ---------------------------------------- |
| B+ 树 | 多路树 | 节点分裂/合并，保证所有叶子节点在同一层（百万级数据仅 3 层）         | 磁盘、大数据、范围查询                              |
| 红黑树  | 二叉树 | 染色 + 旋转，保证左右子树高度差不超过 2 倍（大致平衡）           | 内存、小数据、单点查询（Java TreeMap/HashMap 链表转红黑树） |
| 其他   | -   | BST（最坏退化成链表）、AVL（高度差≤1，严格平衡）、Hash（仅等值查询） | -                                        |


- B+ 树叶子节点容量由 InnoDB 页大小（16KB）决定。

### 2. InnoDB 索引结构


| 索引类型     | 结构（B+ 树）                  | 查询特点                                             |
| -------- | ------------------------- | ------------------------------------------------ |
| 聚簇索引（主键） | 非叶子节点：主键索引；叶子节点：主键 + 完整数据 | 主键查询无需回表，速度极快                                    |
| 二级索引     | 非叶子节点：索引值；叶子节点：索引值 + 主键值  | 查询非索引字段需回表；分散主键回表时，MySQL 会通过 MRR 排序主键后批量回表，减少 IO |


### 3. 联合索引核心规则

#### （1）最左前缀匹配原则

- 核心逻辑：联合索引 `(column1, column2, column3)` 的前缀为 `(column1)(column1, column2)(column1, column2, column3)`（1 个联合索引等价 3 个索引）；  
- 匹配终止：遇到范围查询`><`）停止匹配`>=<=BETWEEN`、前缀匹配 `LIKE` 不终止；  
- 最佳实践：区分度高的字段放在最左侧，过滤更多数据。

#### （2）索引下推（ICP）

- 特性：MySQL 5.6+ 默认开启，使用联合索引的多个字段直接进行多条件判断，减少回表次数；  
- 场景示例（联合索引 `(a,b,c)`）：


| 查询语句                  | 是否走索引 | 说明                                       |
| --------------------- | ----- | ---------------------------------------- |
| `a=1 AND c=1`         | 部分走   | 仅 `a=1` 走索引`c=1` 依赖索引下推过滤                |
| `c=1`                 | 不走    | 无最左列 `a`，违反最左前缀原则                        |
| `b=1 AND c=1`         | 不走    | 无最左列 `a`，违反最左前缀原则                        |
| `b=1 AND a=1 AND c=1` | 全走    | 优化器重排序条件为 `a=1 AND b=1 AND c=1`，匹配完整联合索引 |


#### （3）索引失效核心场景

- 违反最左匹配原则（无最左列/跳过左侧列）；  
- 范围查询后列无法精准匹配（仅能通过索引下推过滤）；  
- 其他：函数/运算操作索引列、使用 `!=NOT INIS NULL`（部分场景）。

### 4. 索引优化最佳实践

- **覆盖索引**：避免 `select` *，仅查询索引包含字段（JPA 可通过 Projection 实现）；*  
*-* *MRR 优化**：依赖 MySQL 自动排序分散主键，减少回表 IO；  
- **联合索引设计**：高频查询字段放左侧，区分度高的字段放左侧。

### 5. SQL 执行计划核心列说明


| 列名            | 含义                                                            |
| ------------- | ------------------------------------------------------------- |
| id            | SELECT 查询的序列标识符                                               |
| select_type   | SELECT 关键字对应的查询类型（SIMPLE/PRIMARY/SUBQUERY 等）                  |
| table         | 用到的表名                                                         |
| partitions    | 匹配的分区，未分区表为 NULL                                              |
| type          | 表的访问方法（性能从优到差：system > const > eq_ref > ref > range > ALL）    |
| possible_keys | 可能用到的索引                                                       |
| key           | 实际用到的索引                                                       |
| key_len       | 所选索引的长度（越短越好，可判断联合索引匹配列数）                                     |
| ref           | 索引等值查询时，与索引比较的列/常量                                            |
| rows          | 预计读取行数（越少越好）                                                  |
| filtered      | 按条件过滤后留存记录的百分比（越高越好）                                          |
| Extra         | 附加信息（Using index=覆盖索引；Using where=过滤；Using filesort=文件排序，需优化） |


## 四、JPA 最佳实践

1. **关联查询优化**`@ManyToOne` 必须使用 `FetchType.LAZY`，避免 N+1 查询；
2. **性能优化**：优先使用 Projection 实现覆盖索引，减少回表；
3. **生产环境配置**：关闭 `open-in-view`，避免长事务占用连接池；
4. **批量操作优化**：开启批量查询/批量抓取`fetch.batch_size`），减少 IO 次数。
